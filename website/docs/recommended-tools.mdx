dux runs your real CLI in a real terminal, with no protocol layer in between. So
whatever providers, MCP servers, and skills you already use keep working inside a
dux agent exactly as they do in your shell. Everything below is hand-picked to
genuinely pair well with a multi-agent dux workflow.

## Tools

The CLIs that power agents, plus the tooling dux leans on. The four defaults ship
preconfigured; any CLI with an interactive and a headless mode can be added (see
[Custom CLI Agents](/docs/custom-agents)).

## MCP servers

Model Context Protocol servers extend what an agent can reach: repositories,
browsers, live docs, and more. Configure them in your CLI the way you normally
would, and they work unchanged inside a dux agent.

<ToolGrid>
  <ToolCard
    name="GitHub MCP Server"
    repo="github/github-mcp-server"
    description="The official GitHub MCP server: issues, PRs, and code search from inside the agent."
  />
  <ToolCard
    name="Context7"
    repo="upstash/context7"
    description="Pulls up-to-date, version-accurate library docs into the agent's context."
  />
  <ToolCard
    name="Playwright MCP"

## Skills

Skills are reusable instructions and workflows your CLI loads on demand (for
example, Claude Code's Agent Skills). Because dux launches the real CLI, your
skills apply per agent and per worktree, exactly as they do elsewhere.
